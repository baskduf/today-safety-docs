# Frontend User Flow And Wireframes

## 1. 설계 목표

이 문서의 목표는 Today Safety 프론트엔드의 핵심 화면을 `작업 전 브리핑 생성 도구` 기준으로 고정하는 것이다.

중심 질문은 하나다.

`현장관리자가 3분 안에 입력하고, 1분 안에 읽고, 바로 브리핑할 수 있는가?`

## 2. 핵심 사용자 흐름

### 기본 흐름

1. 사용자가 오늘 작업 정보를 입력한다.
2. 시스템이 위험 브리핑 초안을 생성한다.
3. 사용자가 위험요약과 사고유형 Top 3를 먼저 읽는다.
4. 사용자가 체크리스트와 근거 사례를 확인한다.
5. 사용자가 브리핑 문안을 복사하거나 수정한다.
6. 관리자가 최종 승인 또는 부적합 처리한다.

### 화면 흐름

```text
/new
  -> 입력 완료
  -> 생성 요청
  -> /result/:id
  -> 검토/수정/승인
  -> PDF 저장 또는 기록 보관
```

## 3. 페이지 구조

### `/new`

새 브리핑 생성의 메인 화면이다.

화면 목적:

- 필수 입력만 빠르게 받는다.
- 선택 입력은 접거나 펼치게 한다.
- 사용자가 첫 진입에서 부담을 느끼지 않게 한다.

기본 섹션:

- 상단 헤더
  - 서비스명
  - 오늘 날짜
  - `새 브리핑`, `브리핑 기록`, `설정`
- 입력 폼
  - 산업군
  - 작업 유형
  - 작업 장소
  - 추가 설명
- 선택 입력 접힘 섹션
  - 고소작업 여부
  - 중장비 사용
  - 화학물질 사용
  - 날씨 반영
  - 작업 인원
  - 작업 시간
- CTA
  - `위험 브리핑 생성하기`

### `/result/:id`

브리핑 초안 확인과 검토의 중심 화면이다.

결과 노출 순서:

1. 오늘의 위험요약
2. 주요 사고유형 Top 3
3. 작업 전 체크리스트
4. 현장 브리핑 문안
5. 유사 재해사례 근거
6. 검토 상태 및 저장 액션

### `/history`

과거 브리핑 목록 화면이다.

목록 컬럼 예시:

- 생성일시
- 산업군
- 작업 유형
- 작업 장소
- 위험등급 또는 위험 키워드
- 검토 상태
- 최종 수정자

### `/history/:id`

승인 이력과 이전 결과를 다시 확인하는 화면이다.

포함 항목:

- 당시 입력값
- 생성 결과 스냅샷
- 승인 상태
- 수정 이력
- PDF 내보내기 기록

### `/settings`

사업장 단위 기본 설정 화면이다.

초기 범위:

- 기본 산업군
- 기본 위치
- 날씨 자동 조회 사용 여부
- PDF 머리말에 표시할 사업장명

## 4. 입력 화면 설계

### 필수 입력

- 산업군
- 작업 유형
- 작업 장소
- 추가 설명

### 선택 입력

- 고소작업 여부
- 중장비 사용
- 화학물질 사용
- 날씨
- 작업 인원
- 작업 시간

### 입력 UX 규칙

- 첫 화면에서는 필수 입력만 기본 노출한다.
- 작업 유형은 자유 입력 + 추천 태그 혼합형으로 둔다.
- 추가 설명은 자유 서술을 허용하되 2~3문장 이내 예시를 보여준다.
- 날씨는 `현재 위치 날씨 불러오기`와 수동 선택을 같이 둔다.
- 화학물질과 중장비는 복수 선택을 허용한다.

### 추가 설명 예시

```text
오늘은 강풍 예보가 있고, 실외 철골 작업 중 크레인 자재 인양이 예정되어 있습니다.
```

## 5. 결과 화면 설계

### 5.1 오늘의 위험요약

가장 먼저 읽히는 영역이다.

표시 원칙:

- 3문장 이내
- 자연어 중심
- 위험 원인과 우선 조치를 같이 제시
- 과장된 확률 표현은 사용하지 않음

### 5.2 주요 사고유형 Top 3

카드 3개로 고정한다.

각 카드 구성:

- 사고유형명
- 원인 2~3개
- 우선 조치 1개
- 관련 근거 열기 버튼

### 5.3 작업 전 체크리스트

현장 활용성이 가장 높은 구간이다.

표시 원칙:

- 체크박스 사용
- 한 줄에 한 항목
- 행동문 형태 유지
- 각 항목에 `근거` 버튼 제공

### 5.4 브리핑 문안

핵심 결과물이다.

표시 원칙:

- 현장에서 읽기 쉬운 짧은 문단
- 줄바꿈이 살아 있는 형태
- `복사`, `PDF 저장`, `수정`, `승인` 버튼 고정

### 5.5 유사 재해사례 근거

기본은 접힘 상태로 둔다.

각 사례 구성:

- 제목
- 유사도 또는 관련도
- 사고유형 태그
- 2~3줄 요약
- 출처
- 원문 링크

### 5.6 검토 상태

상태값:

- 초안
- 수정 필요
- 승인 완료
- 부적합

권장 액션:

- `수정 후 저장`
- `승인`
- `부적합 처리`

주의 문구:

```text
본 브리핑은 AI가 생성한 초안이며, 최종 검토와 현장 적용 책임은 관리자에게 있습니다.
```

## 7. 모바일 와이어프레임

모바일은 한 화면에 모든 내용을 넣지 않고 단계형으로 나눈다.

```text
1. 작업 입력
2. 위험요약
3. 체크리스트
4. 브리핑 문안
5. 근거/승인
```

모바일 우선순위:

- 브리핑 문안 가독성
- 체크리스트 터치 영역
- 근거 펼침/접힘 간결성
- 승인 버튼 접근성

## 8. 예외 상태 설계

### 로딩 상태

- 입력 폼 우측 결과 패널에 스켈레톤 노출
- `사례 검색 중`, `브리핑 문안 생성 중` 같은 단계 문구 표시

### 빈 상태

- 첫 진입 시 결과 패널에는 예시 브리핑 카드 또는 사용 안내 표시

### 오류 상태

- API 오류 시 입력값은 유지
- `다시 생성하기` 버튼 제공
- 날씨 조회 실패는 전체 실패로 보지 않고 수동 입력으로 전환

## 9. MVP 우선순위

### 1순위

- 작업 입력 폼
- 위험요약 카드
- 사고유형 Top 3 카드
- 체크리스트
- 브리핑 문안
- 근거 사례 리스트
- 복사 버튼

### 2순위

- PDF 저장
- 승인 상태
- 히스토리 목록
- 날씨 자동 조회
- 화학물질 검색

### 3순위

- 통계 대시보드
- 관리자 계정 관리
- 사업장별 템플릿
- 팀 공유
- PWA 고도화
