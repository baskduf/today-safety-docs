# Frontend Component And State Spec

## 1. 목적

이 문서는 Today Safety 프론트엔드 MVP에서 필요한 핵심 컴포넌트와 상태값을 정의한다.

목표는 컴포넌트를 많이 만드는 것이 아니라, `브리핑 생성 -> 검토 -> 저장` 흐름을 안정적으로 구현하는 것이다.

## 2. 핵심 컴포넌트 목록

### `BriefInputForm`

역할:

- 오늘 작업 정보를 입력받는다.
- 필수 입력과 선택 입력을 구분한다.

포함 필드:

- 산업군
- 작업 유형
- 작업 장소
- 추가 설명
- 고소작업 여부
- 중장비
- 화학물질
- 날씨 반영 방식
- 작업 인원
- 작업 시간

UX 규칙:

- 기본 노출은 필수 입력 중심
- 선택 입력은 아코디언 또는 접힘 패널 사용
- 제출 버튼은 항상 폼 하단 고정

### `RiskSummaryCard`

역할:

- 생성 결과의 첫 문장을 보여준다.

표시 항목:

- 위험요약 제목
- 요약 본문
- 주요 위험 태그

UX 규칙:

- 가장 위에 둔다.
- 본문 길이는 최대 3문장 기준으로 잘라낸다.

### `AccidentTypeCardList`

역할:

- 주요 사고유형 Top 3를 보여준다.

각 카드 항목:

- 사고유형명
- 원인
- 우선 조치
- 근거 보기 버튼

UX 규칙:

- 3개 카드를 우선 고정
- 각 카드는 위험 강조색을 약하게만 사용

### `ChecklistPanel`

역할:

- 작업 전 점검 항목을 체크박스 형태로 제공한다.

각 항목:

- 점검 문장
- 근거 버튼
- 선택 시각 또는 사용자 체크 상태

UX 규칙:

- 모바일에서도 한 줄 클릭이 쉬워야 한다.
- 체크 상태는 세션 중 유지한다.

### `BriefingDraftPanel`

역할:

- 현장 브리핑 문안을 보여주고 편집하게 한다.

포함 기능:

- 문안 본문
- `복사`
- `PDF 저장`
- `수정`
- `승인`

UX 규칙:

- 읽기 모드와 편집 모드를 분리한다.
- 편집 모드에서는 텍스트영역과 `수정 후 저장` 버튼을 제공한다.

### `EvidenceAccordion`

역할:

- 근거 사례를 접힘 구조로 보여준다.

각 사례 항목:

- 제목
- 관련 사고유형
- 유사도
- 요약
- 출처
- 원문 링크

UX 규칙:

- 기본은 접힘
- 처음부터 원문 전체를 펼치지 않는다
- 근거가 없는 항목은 `근거 준비 중`으로 명시

### `ReviewStatusBar`

역할:

- 검토 상태와 책임 경계 문구를 표시한다.

상태:

- 초안
- 수정 필요
- 승인 완료
- 부적합

필수 문구:

```text
본 브리핑은 AI가 생성한 초안이며, 최종 검토와 현장 적용 책임은 관리자에게 있습니다.
```

### `ResultActionBar`

역할:

- 결과 화면의 주요 액션을 모은다.

포함 버튼:

- 복사
- PDF 저장
- 수정
- 승인
- 부적합 처리

## 3. 페이지 상태 모델

### 생성 화면 상태

- `idle`
  - 입력 전 기본 상태
- `submitting`
  - 생성 요청 진행 중
- `submit_failed`
  - 생성 실패
- `result_ready`
  - 결과 수신 완료

### 결과 문서 상태

- `draft`
  - 생성 직후 초안
- `editing`
  - 관리자 수정 중
- `approved`
  - 승인 완료
- `rejected`
  - 부적합 처리

### 근거 로딩 상태

- `collapsed`
- `loading`
- `ready`
- `error`

## 4. 디자인 톤 가이드

서비스 성격은 `공공서비스 느낌 + 현장 업무도구 느낌`에 가깝게 유지한다.

지양할 것:

- 채팅 버블 중심 레이아웃
- 과도하게 화려한 AI 브랜드 표현
- 감성형 일러스트 중심 화면

권장 방향:

- 단정한 카드와 표면 구분
- 선명한 상태 색상
- 강한 대비의 타이포그래피
- 현장용 문서에 가까운 정보 구조

## 5. 색상 역할 가이드

- 기본 배경: 밝은 회색 또는 흰색
- 기본 텍스트: 진한 남색 또는 거의 검은색
- 정보: 파랑
- 정상/확인: 초록
- 주의: 노랑 또는 주황
- 위험: 빨강

원칙:

- 빨강은 정말 중요한 중지 조건이나 위험 상태에만 사용
- 전체 화면을 경고색으로 채우지 않음

## 6. 타이포그래피 가이드

- 제목은 짧고 단호하게 쓴다.
- 위험요약은 문단형으로 읽히게 한다.
- 체크리스트는 질문형 또는 행동형 한 문장으로 통일한다.
- 브리핑 문안은 관리자 낭독을 고려해 줄바꿈을 살린다.

## 7. UX 문구 가이드

권장:

- `오늘의 위험요약`
- `작업 전 체크리스트`
- `현장 브리핑 문안`
- `근거 사례 보기`
- `수정 후 저장`

비권장:

- `LLM 응답`
- `프롬프트`
- `AI 채팅`
- `모델 결과`

## 8. 접근성 및 현장 사용성

- 모바일에서 버튼 높이는 충분히 크게 유지한다.
- 체크박스와 근거 버튼은 장갑 낀 손가락도 누르기 쉬운 크기로 잡는다.
- 긴 문장은 자동 줄바꿈되게 하고, 스크롤 영역을 과도하게 중첩하지 않는다.
- PDF 저장 전에도 화면에서 바로 읽을 수 있어야 한다.

## 9. 구현 우선순위

### 다음 단계

- `ReviewStatusBar`
- `ResultActionBar`
- `HistoryList`

- 팀 공유용 링크 패널
