# 기술 구성 및 아키텍처 정리

## 1. 추천 프레임워크

이 아이디어에는 아래 구성이 가장 적합하다.

- 프론트엔드: `Next.js + TypeScript`
- 백엔드: `FastAPI + Pydantic`
- AI 연동: `Python 서비스 레이어 + OpenAI Responses API`
- 데이터베이스: `PostgreSQL`

이 조합을 추천하는 이유는 다음과 같다.

- Next.js는 데모 제작 속도가 빠르고 UI 구축이 쉽다.
- FastAPI는 API 문서화와 스키마 검증이 강하다.
- Python은 데이터 처리, 검색 파이프라인, AI 연동, 모델 실험에 유리하다.
- Pydantic은 입력값 검증과 응답 구조 정리에 적합하다.
- OpenAI Responses API는 구조화된 생성 결과를 안정적으로 다루기 좋다.

## 2. 프론트엔드 / 백엔드 / AI를 나눠야 하는가

논리적으로는 나누는 것이 맞지만, 처음부터 물리적으로 3개 서비스로 분리하는 것은 권장하지 않는다.

### 권장 구조

- 프론트엔드: 별도 애플리케이션
- 백엔드: 별도 API 서버
- AI: 백엔드 내부의 서비스 모듈

즉, 배포 단위는 우선 2개로 가져간다.

1. Next.js 프론트엔드
2. FastAPI 백엔드

AI는 독립 서버가 아니라 FastAPI 안의 내부 계층으로 둔다.

## 3. 왜 AI를 별도 서비스로 바로 떼지 않는가

- 공모전 단계에서는 속도와 단순성이 중요하다.
- 3개 서비스로 쪼개면 인증, 네트워크, 로깅, 배포가 복잡해진다.
- 현재 아이디어의 AI는 독립 플랫폼보다 백엔드 기능의 일부에 가깝다.
- 모델 실험과 프롬프트 수정이 잦은 초기 단계에서는 Python 백엔드 내부에 두는 편이 유리하다.

## 4. 권장 시스템 구조

### 프론트엔드

- 작업 조건 입력 폼
- 결과 대시보드
- 위험요약, 체크리스트, 브리핑 표시
- PDF 저장 또는 공유 기능

### 백엔드

- 요청 수신 및 검증
- 공공데이터 수집 및 조회
- 날씨 데이터 조회
- 유사 사례 검색
- 위험도 및 사고유형 산출
- AI 호출 오케스트레이션
- 승인/수정 후 승인/부적합 흐름 관리
- 감사로그 및 출력 버전 이력 관리

### AI 레이어

- 검색 질의 구성
- 유사 사고사례 요약
- 위험설명 생성
- 체크리스트 생성
- 브리핑 문안 생성

## 5. 권장 저장 구조

초기 MVP에서도 저장 계층은 역할별로 구분해 두는 편이 좋다.

- 원천 파일/첨부 원문
  - PDF, API 원문 응답, 재현용 원천 데이터는 object storage 성격의 저장소에 보관
- 정규화 메타/운영 데이터
  - 요청 이력, 정규화 테이블, 승인 결과, 버전 정보는 PostgreSQL에 저장
- 검색 계층
  - 1차는 PostgreSQL 기반 메타 조회로 시작하되, 후속 단계에서 BM25/embedding 인덱스 레이어로 분리

이 구조는 원문 보존, 운영 조회, 검색 성능 최적화를 서로 분리하는 데 유리하다.

## 6. 추천 데이터 흐름

1. 사용자가 작업 조건 입력
2. 백엔드가 요청값 검증
3. 공공데이터와 날씨 데이터 조회
4. 검색 및 경량 예측 계층이 유사 사고유형과 위험요인을 정리
5. 계산 결과와 참고 데이터를 생성형 AI에 전달
6. AI가 위험요약, 체크리스트, 브리핑 문안 생성
7. 현장관리자 또는 승인자가 결과를 검토·수정·승인
8. 승인 이력과 출력 버전을 저장하고 결과를 프론트엔드에 반환

## 7. 실시간과 배치 처리 구분

실시간성이 필요한 계층과 배치 적재 계층은 구분하는 편이 좋다.

- 실시간 또는 짧은 캐시
  - 날씨, MSDS, 최신 재해사례 조회
- 배치 적재 및 색인
  - 승인 데이터, SIF 아카이브, taxonomy, 첨부 원문 색인 갱신

이 구분을 해 두면 운영 비용과 응답속도를 동시에 관리하기 쉽다.

## 8. MVP 기준 기술 판단

초기 MVP에서는 아래 구조가 가장 현실적이다.

- 규칙 계층: 작업중지 검토 조건, 법정/사내 차단조건 상단 배치
- 경량 예측 모델: 사고유형 Top N, 상대적 위험등급 도출
- 검색 계층: 유사 재해사례, SIF, MSDS, 날씨 위험요인 검색
- 생성형 AI: 위험요약, 설명 문장, 체크리스트, 브리핑 생성

즉, `규칙 + 검색 + 생성형 AI + 경량 분류`의 하이브리드 구조가 가장 적합하다. 이 구조는 정밀 사고확률 예측보다 설명가능성과 운영 안정성을 우선하는 방향과도 맞는다.

## 9. Human-in-the-Loop와 감사 요건

이 서비스는 법적 위험성평가를 자동 대체하는 시스템이 아니므로, 사람 검토를 전제로 한 운영 구조가 필요하다.

- 결과 상태는 `승인`, `수정 후 승인`, `부적합` 중 하나로 기록
- 입력값, 근거 citation, 출력 버전, 승인자, 수정 이력을 함께 저장
- 작업 전 브리핑 결과는 추후 재검토와 품질개선에 사용할 수 있도록 감사로그로 남김

이 기록은 운영 품질 관리뿐 아니라 후속 개인화와 모델 개선에도 직접 활용된다.

## 10. 비추천 구조

아래 방식은 초기 단계에서 비효율적일 가능성이 높다.

- 프론트엔드 / 백엔드 / AI를 모두 독립 마이크로서비스로 시작
- 모든 기능을 LLM 한 번 호출로 해결하려는 구조
- 정밀 사고확률 예측을 MVP 핵심 기능으로 설정하는 구조
- 공공데이터만으로 법적 위험성평가 자동판정을 전제하는 구조

## 11. 최종 권장안

가장 추천하는 최종 구성은 다음과 같다.

- 프론트엔드: Next.js
- 백엔드: FastAPI
- AI: FastAPI 내부 Python 모듈
- DB: PostgreSQL
- 외부 AI: OpenAI Responses API

한 줄로 요약하면, `프론트엔드와 백엔드는 분리하고 AI는 백엔드 내부 계층으로 두되, 예측·검색·생성을 결합한 하이브리드 구조`가 가장 현실적이다.
