# 기술 구성 및 아키텍처 정리

## 1. 추천 프레임워크

이 아이디어에는 아래 구성이 가장 적합하다.

- 프론트엔드: `Next.js + TypeScript`
- 백엔드: `FastAPI + Pydantic`
- AI 연동: `Python 서비스 레이어 + OpenAI Responses API`
- 데이터베이스: `PostgreSQL`

이 조합을 추천하는 이유는 다음과 같다.

- Next.js는 데모 제작 속도가 빠르고 UI 구축이 쉽다.
- FastAPI는 API 문서화와 스키마 검증이 강하다.
- Python은 데이터 처리, AI 연동, 모델 실험에 유리하다.
- Pydantic은 입력값 검증과 응답 구조 정리에 적합하다.
- OpenAI Responses API는 생성형 AI 호출 구조를 단순하게 가져가기 좋다.

## 2. 프론트엔드 / 백엔드 / AI를 나눠야 하는가

논리적으로는 나누는 것이 맞지만, 처음부터 물리적으로 3개 서비스로 분리하는 것은 권장하지 않는다.

### 권장 구조

- 프론트엔드: 별도 애플리케이션
- 백엔드: 별도 API 서버
- AI: 백엔드 내부의 서비스 모듈

즉, 배포 단위는 우선 2개로 가져간다.

1. Next.js 프론트엔드
2. FastAPI 백엔드

AI는 독립 서버가 아니라 FastAPI 안의 내부 계층으로 둔다.

## 3. 왜 AI를 별도 서비스로 바로 떼지 않는가

- 공모전 단계에서는 속도와 단순성이 중요하다.
- 3개 서비스로 쪼개면 인증, 네트워크, 로깅, 배포가 복잡해진다.
- 현재 아이디어의 AI는 독립 플랫폼보다 백엔드 기능의 일부에 가깝다.
- 모델 실험과 프롬프트 수정이 잦은 초기 단계에서는 Python 백엔드 내부에 두는 편이 유리하다.

## 4. 권장 시스템 구조

### 프론트엔드

- 작업 조건 입력 폼
- 결과 대시보드
- 체크리스트 및 브리핑 표시
- PDF 저장 또는 공유 기능

### 백엔드

- 요청 수신 및 검증
- 공공데이터 수집 및 조회
- 날씨 데이터 조회
- 위험도 계산
- 사고유형 추정
- AI 호출 오케스트레이션

### AI 레이어

- 프롬프트 관리
- 유사 사고사례 요약
- 위험설명 생성
- 체크리스트 생성
- 브리핑 문안 생성

## 5. 추천 데이터 흐름

1. 사용자가 작업 조건 입력
2. 백엔드가 요청값 검증
3. 공공데이터와 날씨 데이터 조회
4. 규칙 기반 필터링과 분석 모델로 위험도 및 사고유형 계산
5. 계산 결과와 참고 데이터를 생성형 AI에 전달
6. AI가 체크리스트와 브리핑 문안 생성
7. 결과를 프론트엔드에 반환

## 6. MVP 기준 기술 판단

초기 MVP에서는 아래 구조가 가장 현실적이다.

- 분석 모델: 위험도 등급, 사고유형 Top N 도출
- 룰베이스: 필수 점검 항목, 작업중지 기준
- 생성형 AI: 설명 문장, 체크리스트, 브리핑 생성

즉, `예측 모델 + 룰 + 생성형 AI`의 하이브리드 구조가 가장 적합하다.

## 7. 비추천 구조

아래 방식은 초기 단계에서 비효율적일 가능성이 높다.

- 프론트엔드 / 백엔드 / AI를 모두 독립 마이크로서비스로 시작
- 모든 기능을 LLM 한 번 호출로 해결하려는 구조
- 정밀 사고확률 예측을 MVP 핵심 기능으로 설정하는 구조

## 8. 최종 권장안

가장 추천하는 최종 구성은 다음과 같다.

- 프론트엔드: Next.js
- 백엔드: FastAPI
- AI: FastAPI 내부 Python 모듈
- DB: PostgreSQL
- 외부 AI: OpenAI Responses API

한 줄로 요약하면, `프론트엔드와 백엔드는 분리하고 AI는 백엔드 내부 계층으로 두는 구조`가 가장 현실적이다.
